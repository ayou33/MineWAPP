# AGENTS.md — MineWAPP (SolidJS / TypeScript)

Minimal SolidJS + Vite + TailwindCSS web application template. File-system routing via `@generouted/solid-router`, Web Worker request layer, headless form engine.

## Commands
```bash
pnpm dev                  # Dev server at http://localhost:3000
pnpm build [test|prod]    # Production bundle (via tsx scripts/build.ts)
pnpm zip-worker           # Rebuild after editing public/worker/*.js
pnpm lint:fix             # Auto-fix ESLint issues
pnpm preview              # Preview built assets
```

## Module Anatomy
Every feature lives in `src/module/<feature>/` with conventions:
```
src/module/foo/
├── Foo.index.tsx        # page entry – default export is the page component
├── Foo.module.scss      # scoped styles (optional)
├── foo.api.ts           # API defs using get/post from @/tools/request ONLY
├── foo.helper.ts        # module-specific helpers (optional)
├── store/
│   ├── store.ts         # state type + defaults
│   └── actions.ts       # createScopedActions mutations
├── components/          # module-specific components (optional)
└── hooks/               # module-specific hooks (optional)
```
A new module's state slice must be registered in `src/store/index.ts` (`State` type + initial value).

## Page ↔ Module Pattern
Pages are **thin wrappers**. They import a module's entry component and pass it as a `ValidComponent` to `ScopedPage`:
```tsx
// src/pages/foo.tsx
import ScopedPage from '@/provider/ScopedPage'
import { AUTH_SCOPE } from '@/config'
import Foo from '@/module/foo/Foo.index'

export default function FooPage () {
  return <ScopedPage scope={AUTH_SCOPE.PUBLIC}>{Foo}</ScopedPage>
}
```
**Critical**: `ScopedPage` uses `<Dynamic component={children} />` — children must be a `ValidComponent` function reference, **never JSX** like `<Foo />`.

## Request Layer (`src/tools/request/index.ts`)
- **Never** import axios directly. Use `get`/`post` factory functions.
- Requests run through `lunzi/stateFetch` priority queue with optional Web Worker transport.
- Mock system available via `src/tools/mock/` — mock entries matched by URL prefix.
```ts
export const fetchFoo = get<FooData, FooParams>('/api/foo/data')
export const submitBar = post<Result, BarParams>('/api/bar/submit')
```

## ScopedPage (`src/provider/scopedPage/ScopedPage.tsx`)
Wraps every page. Auto-cancels in-flight requests, toasts, and popups on unmount. Namespaces events to the page lifecycle.

```tsx
// src/pages/foo.tsx
export default function FooPage () {
  return <ScopedPage scope={AUTH_SCOPE.PRIVATE} pageCode="foo">{Foo}</ScopedPage>
}
```

Props:
- `scope` — `AUTH_SCOPE` level (default: `PUBLIC`). Delegated to `PageGuard`.
- `pageCode` — optional string tag for debugging/logging.
- `children` — must be a `ValidComponent` function reference (not JSX).

Context available inside the page component via `usePageContext()`:
```tsx
const { request, toast, popup, interval, delay, on, emit } = usePageContext()
const [fetchData, loading] = request(fetchFoo)
```
`on`/`emit` are namespaced per page instance — events do not leak across pages.

### PageGuard (`src/provider/scopedPage/PageGuard.tsx`)
Embedded inside `ScopedPage`. Handles route-level access control reactively:
- **Unauthenticated + PRIVATE+** → redirects to `AUTH_PATH` (via `useNavigate`).
- **Authenticated but below required scope (AUTHED+)** → renders inline 403 screen.

Guard logic uses `application.role()` (a reactive signal) — re-evaluates on login/logout without a page reload.

```ts

on('LOGIN', handler)

```
Thin wrapper over `lunzi/useEvent`. Page-scoped `on`/`emit` in `ScopedPage` auto-namespace to avoid cross-page leaks.

## Application Singleton (`src/app/application.ts`)
```ts

application.role()        // USER_ROLE enum
```

## Styling Rules

### TailwindCSS v4 — `.css` files only
TailwindCSS v4 uses `@tailwindcss/vite` plugin which **only processes `.css` files**. Never `@import "tailwindcss"` inside SCSS.
- `src/style/tailwind.css` — TailwindCSS entry with `@theme` block. Must remain `.css`.
- Use `@theme` to expose CSS variables as Tailwind utilities.

### SCSS — custom styles only
- `src/style/var.scss` — CSS custom properties (`:root`): theme colors, `--transition-duration`, `--ease-back-in-out`.
- `src/style/transition.scss` — All animation classes (tip animations, popup transitions). Uses CSS variables, **not SCSS variables**, for duration and easing.
- `src/style/index.scss` — `@import "./var.scss"` + `@import "./transition.scss"`, base reset styles.
- Module-scoped styles use `*.module.scss`.

### CSS Variables
All shared timing/easing values live as CSS custom properties in `var.scss`:
```scss
:root {
  --transition-duration: 500ms;
  --ease-back-in-out: cubic-bezier(0.68, -0.25, 0.2, 1.2);
}
```
Read them in TypeScript via `getComputedStyle(document.documentElement).getPropertyValue('--transition-duration')`.

## Animation Conventions

### Tips (toasts)
Self-managed animation in `Tip.tsx` via CSS class toggling — **no `<Transition>` wrapper**. `<Transition>` inside `<For>` breaks exit animations because `<For>` destroys DOM before `<Transition>` can animate.
- Three position-specific classes: `.tip-slide-down` (TOP), `.tip-scale` (CENTER), `.tip-slide-up` (BOTTOM).
- Enter: toggle `.tip-visible` after double-`requestAnimationFrame`.
- Exit: remove `.tip-visible`, delay removal by `--transition-duration`.
- Swipe-to-dismiss only for TOP position.

### Popups
Use `<TransitionGroup>` from `solid-transition-group` for list-based enter/exit.
`forceRemove` is a pure teardown timeout based on `--transition-duration` + safety margin — it never manipulates the closing array directly.

## A/B Test System (`src/features/abTest/`)
Group-based feature flag system for conditional rendering and lazy loading.

| File | Purpose |
|---|---|
| `ABTest.tsx` | Main component — renders children/fallback based on `ifFeatureAllowed` |
| `Alternative.tsx` | Lazy-loads a component via dynamic import, switches between `default`/`fallback` exports by an `Accessor<boolean>` predicate |
| `FeatureAlternative.tsx` | Convenience wrapper around `Alternative` for feature-flag-driven lazy loading |
| `sample-feature.ts` | Feature enum, `ControlledFeatures` config, and version compatibility helpers |

**Usage — static children:**
```tsx
import ABTest from '@/features/abTest/ABTest'
import { Feature } from '@/features/abTest/sample-feature'

<ABTest feature={Feature.A} fallback={<OldUI />}>
  <NewUI />
</ABTest>
```

**Usage — lazy/dynamic load:**
```tsx
<ABTest
  feature={Feature.A}
  load={() => import('./MyFeature')}  // module must export default + optional fallback
/>
```

**Feature configuration** (`sample-feature.ts`):
```ts
export const ControlledFeatures: Partial<Record<Feature, FeatureControl>> = {
  [Feature.A]: {
    groups: [1, 2],          // only users in these groups see the feature
    versions: { ios: '2.0.0', android: '2.0.0', web: true },
  },
}
```
- `groups`: if omitted, all authenticated users qualify.
- `versions`: version-gate for native app builds; `web: true` enables on web.
- `ifFeatureAllowed(feature, userGroup)` — pure helper, safe to call outside components.
- Add new features to the `Feature` enum and optionally to `ControlledFeatures`.

## Form System (`src/features/form/`)
Headless reactive form engine:
- `types.ts` — `Rule`, `FormConfig`, `CascadeContext`, `Relation`, `FieldMeta`, `ValidateOn`
- `rules.ts` — Built-in validators: `required`, `minLength`, `maxLength`, `min`, `max`, `pattern`, `email`, `match`, `oneOf`, `custom`
- `createForm.ts` — Reactive stores, cascade with loop protection, relation effects, bind helpers (`bind`, `bindNumber`, `bindCheckbox`, `bindSelect`), `validate`/`submit`/`reset`

## Form Components (`src/components/form/`)
Themed, reusable form UI that integrates directly with the `createForm` headless engine.

| Component | File | Purpose |
|---|---|---|
| `FormItem` | `FormItem.tsx` | Layout wrapper: optional label (with required `*`), children slot, error message with `AlertCircle` icon, helper text |
| `Input` | `Input.tsx` | MD3 outlined text input; accepts `prefix`/`suffix` JSX slots; spread-compatible with `form.bind()` |
| `PasswordInput` | `PasswordInput.tsx` | Wraps `Input`; manages show/hide state internally; Eye/EyeOff suffix toggle |

All three are re-exported from `@/components/form`.

### Typical usage
```tsx
import { FormItem, Input, PasswordInput } from '@/components/form'
import createForm from '@/features/form/createForm'
import { required, minLength } from '@/features/form/rules'

const form = createForm({
  initialValues: { email: '', password: '' },
  rules: {
    email:    [required('请输入邮箱')],
    password: [required('请输入密码'), minLength(6, '密码至少 6 位')],
  },
  validateOn: 'blur',
})

// In JSX:
<FormItem label="邮箱" required error={form.error('email')}>
  <Input {...form.bind('email')} type="email" placeholder="your@email.com"
    prefix={<Icon name="Mail" size={16} />} error={form.error('email')} />
</FormItem>
<FormItem label="密码" required error={form.error('password')}>
  <PasswordInput {...form.bind('password')}
    prefix={<Icon name="Lock" size={16} />} error={form.error('password')} />
</FormItem>
```

### Input styling notes
- Uses `--c-input-outlined-*` CSS tokens from `var.scss` (MD3 themed, light + dark).
- Error state activates when `error` prop is truthy; border becomes `--c-input-outlined-border-error`.
- Focus ring uses `--state-focus` CSS variable (also `var.scss`).
- `Input` uses `splitProps` — passing `bind()` spread and extra HTML attributes is safe.

### PasswordInput note
- `type` is managed internally; do not pass `type` to `PasswordInput`.
- `prefix` is forwarded to `Input`; `suffix` is the internal toggle button and cannot be overridden.

## Design System — MD3 Theme (Light & Dark)

### Overview

This project uses a **two-layer Material Design 3 token system**:

| Layer | Prefix | Purpose |
|---|---|---|
| MD3 Color Roles | `--md-*` | Primitive palette values per theme |
| Semantic Component Tokens | `--c-*` | Purpose-specific, map to MD3 roles |

Dark mode is activated by adding `class="dark"` to `<html>`. All `--md-*` and most `--c-*` tokens are overridden in `.dark {}` in `src/style/var.scss`.

---

### Token Files

| File | Role |
|---|---|
| `src/style/var.scss` | Source of truth — defines all `--md-*` and `--c-*` CSS custom properties for both themes + scrollbar styles |
| `src/style/tailwind.css` | Exposes every `--c-*` token as a Tailwind utility via `@theme inline` (e.g. `bg-c-surface`, `text-c-text-muted`) |

### Coding Rules

#### ✅ DO — use semantic tokens via Tailwind or CSS vars
```tsx
// Background
<div class="bg-c-surface border border-c-outline rounded-xl">

// Text hierarchy
<h2 class="text-c-text">Title</h2>
<p class="text-c-text-muted">Supporting</p>
<span class="text-c-text-subtle">Metadata</span>

// Button filled
<button class="bg-c-btn-filled-bg text-c-btn-filled-text hover:bg-c-btn-filled-hover-bg">

// Input filled
<input class="bg-c-input-bg text-c-input-text placeholder:text-c-input-placeholder" />

// Shadow via CSS var (inline style)
<div style={{ "box-shadow": "var(--md-shadow-md)" }}>
```

#### ❌ DON'T — hardcode colors that should adapt to theme
```tsx
// ❌ breaks dark mode
<div class="bg-white text-gray-800 border-gray-200">
<button class="bg-blue-500 text-white">

// ✅ use semantic token instead
<div class="bg-c-surface text-c-text border-c-outline">
<button class="bg-c-btn-filled-bg text-c-btn-filled-text">
```

#### Exception — intentional dark/branded sections
```tsx
// ✅ OK: intentional full-bleed dark gradient (hero, CTA, footer)
//    These are branded design decisions, not theme-sensitive surfaces.
<section style={{ background: 'linear-gradient(160deg, #050e2a ...)' }}>
<footer class="bg-black text-white">
```

#### Accessing tokens in TypeScript/JS
```ts
const duration = getComputedStyle(document.documentElement)
  .getPropertyValue('--transition-duration').trim() // "500ms"

// Chart series colors
const colors = [1,2,3,4,5,6].map(n =>
  getComputedStyle(document.documentElement).getPropertyValue(`--c-chart-${n}`).trim()
)
```

#### Toggling dark mode
```ts
// Enable dark mode
document.documentElement.classList.add('dark')
// Disable
document.documentElement.classList.remove('dark')
// Read
const isDark = document.documentElement.classList.contains('dark')
```

### MD3 Elevation Model

MD3 uses **tonal color overlay** (not drop shadows alone) to communicate elevation:

| Level | Token | dp | Usage |
|---|---|---|---|
| 0 | `--c-surface` | 0dp | Cards, list items |
| 1 | `--c-surface-1` | 1dp | Sticky nav, table header, FAB |
| 2 | `--c-surface-2` | 3dp | Sidebar, elevated buttons |
| 3 | `--c-surface-3` | 6dp | Modals, drawers, pickers |
| 4 | `--md-surface-4` | 8dp | Menus, popups |
| 5 | `--md-surface-5` | 12dp | Tooltips |

Always pair elevation with the matching `--md-shadow-*` for the drop shadow component:
```tsx
// Card at elevation 1
<div class="bg-c-surface-1" style={{ "box-shadow": "var(--md-shadow-sm)" }}>

// Dialog at elevation 3
<div class="bg-c-surface-2" style={{ "box-shadow": "var(--md-shadow-dialog)" }}>
```

### State Layers (MD3)

MD3 uses translucent color overlays on interactive elements for hover/focus/pressed states:

| State | Token | Opacity |
|---|---|---|
| Hover | `--state-hover` | 8% primary |
| Focus | `--state-focus` | 12% primary |
| Pressed | `--state-pressed` | 16% primary |
| Disabled | `--state-disabled` | 12% on-surface |

```tsx
// Apply state layer as a pseudo-element or background
<button class="relative overflow-hidden hover:bg-[var(--state-hover)]">
```

### Don't Add New Color Variables to `tailwind.css`

All new design tokens belong in `var.scss`. The `@theme inline` block in `tailwind.css` only mirrors `--c-*` tokens as Tailwind utilities. Add a new line there only when adding a new `--c-*` semantic token to `var.scss`.
